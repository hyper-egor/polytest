Монолитное Spring Boot приложение для поиска арбитражных возможностей в `negRisk` событиях Polymarket по данным из websocket.

Важно: в рамках тестового задания анализ идёт только по live-stream данным в момент запуска приложения.  
Это демонстрация подхода к обработке и оценке сигналов, а не полноценная статистическая подготовка боевой стратегии (для боевого применения нужен большой накопленный исторический массив данных).

## Что делает приложение

1. Загружает события из Polymarket API.
2. Фильтрует только подходящие события/рынки.
3. Подписывается на websocket CLOB по токенам исходов.
4. Поддерживает локальные стаканы (snapshot + incremental updates).
5. Ищет 2 типа арбитражных сигналов:
   - `YES+NO` внутри одного market.
   - `ALL YES` по всем market внутри одного event.
6. Симулирует вход/выход и считает агрегаты в логах:
   - реализованный PnL,
   - “подозрительный” PnL (слишком быстрый),
   - длительность сигналов,
   - нереализованный капитал/прибыль.

## Принятые допущения

1. **Комиссии (fee):** в симуляции берём только рынки без fee (`market.feesEnabled == false`), чтобы упростить MVP.  
   Важно: если fee есть, его нужно учитывать в каждой покупке/продаже.

2. **Ликвидность/активность (фильтр):**
   - `volume_min >= 100000`,
   - `liquidity_min >= 5000`,
   - `event.enableOrderBook == true`,
   - `event.active == true`, `archived == false`, `closed == false`,
   - `event.enableNegRisk == true`, `event.negRisk == true`,
   - в событии минимум 3 market,
   - каждый market должен иметь 2 outcome (`YES/NO`) и 2 `clobTokenIds`.

3. **Типы сигналов и пороги входа:**
   - `YES+NO`: в одном market покупаем оба исхода, если дельта больше `0.9%`  
     (`1 - (yes_ask + no_ask)`).
   - `ALL YES`: покупаем корзину YES по всем market event, если дельта больше `1.5%`  
     (`1 - sum(all_yes_ask)`).

4. **Без усреднения/докупки:** если позиция уже открыта, повторный вход по этому же market/event не делаем, пока позиция не закроется.

5. **TP (ранний выход):** позиция закрывается до финального исхода события, если можно продать с тем же критическим отклонением, что использовалось для входа:
   - для `YES+NO`: `sell_price >= buy_price * (1 + 0.9%)`,
   - для `ALL YES`: `sell_price >= buy_price * (1 + 1.5%)`.

6. **Проскальзывание не учитывается:** в реальной торговле его нужно добавлять в модель затрат (по аналогии с fee, по эмпирике).

7. **Подозрительный (нереалистичный) профит:** отдельно учитывается прибыль, если сделка закрылась слишком быстро (`< 50 ms`) или до фиксации исчезновения входного сигнала.

8. **API-глюки Polymarket:** не все случаи покрыты (например, дубли market внутри одного event). Часть дублей отсекается по `market.question`.

9. **Метрики по открытым позициям (unrealized):** считаются и логируются:
   - заблокированный капитал (позиции ещё не закрыты),
   - потенциальная прибыль по открытым позициям (ещё не зафиксирована сделкой продажи).

## Идеи на будущее

1. **Конвертация одного `NO` в серию `YES` по прочим исходам (`negRisk`):** идея технически здравая и может дать арбитраж при локальной переоценке `YES`, но execution-риск высокий (глубина стакана, частичные исполнения, сдвиг цены во время набора/выхода).
2. **Генерация `YES` через on-chain mint/split за `1 USDC` при переоценке `YES` в стакане:** идея также здравая как арбитражный контур, но требует аккуратной оценки комиссий/латентности/риска исполнения; без этого может оказаться убыточной даже при видимом ценовом отклонении.

## Где какая логика в коде

- [src/main/java/ru/polytest/StakanService.java](src/main/java/ru/polytest/StakanService.java)  
  Инициализация пайплайна, фильтрация событий, формирование списка токенов, обработка websocket событий, периодическая печать общей статистики.

- [src/main/java/ru/polytest/PolyService.java](src/main/java/ru/polytest/PolyService.java)  
  REST-загрузка событий/рынков Polymarket (`events/keyset`, `markets/keyset`) и фильтры “торгуемости”.

- [src/main/java/ru/polytest/PolyHandler.java](src/main/java/ru/polytest/PolyHandler.java)  
  WebSocket подключение к `wss://ws-subscriptions-clob.polymarket.com/ws/market`, подписка на токены, парсинг `book` и `price_change`.

- [src/main/java/ru/polytest/Book.java](src/main/java/ru/polytest/Book.java)  
  Локальный стакан: snapshot (`book`) и инкрементальные апдейты (`price_change`).

- [src/main/java/ru/polytest/Instrument.java](src/main/java/ru/polytest/Instrument.java)  
  Основная торговая логика: поиск сигналов, виртуальный вход, попытка закрытия по TP, расчёт PnL и lifetime сигналов.

## Что смотреть в логах

Ключевые строки:
- `1 OUTCOME (YES+NO)` / `CLOSE 1 OUTCOME (YES+NO)`,
- `ALL YES` / `CLOSE ALL YES`,
- блоки `total ...` (агрегированная статистика раз в минуту),
- `total ... profit (HIGH RISK <50ms)` — подозрительный PnL.

## Запуск

```bash
./gradlew bootRun
```

Приложение стартует как обычный Spring Boot сервис и сразу начинает сбор/обработку данных.
